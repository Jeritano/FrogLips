## MEDIUM severity

### [bug] Dry-run edit_file/multi_edit preview diverges from real executor (silent first-match replace vs. multi-match reject)
- **Where:** `src/lib/agent-loop/dry-run.ts:148-151, 181-190`  ·  area: ts-agent-loop-rest  ·  confidence: high
- **Problem:** The dry-run preview for edit_file and multi_edit computes the 'after' text in JS with `before.replace(oldStr, newStr)` (single, first-occurrence replace) when replace_all is false. The actual Rust executor (src-tauri/src/agent/fs.rs:932-947, 1315-1325) first counts matches and REJECTS the call with an error (`old_string matches {count} times; pass replace_all=true...`) when count>1 and replace_all is false. So when old_string occurs multiple times, the dry-run shows a clean, successful single-occurrence diff, but the real run would error out. The preview misrepresents what the approved operation actually does — the exact thing dry-run exists to prevent. Conversely, JS `String.replace` replacing the first match silently also hides the ambiguity the real tool surfaces.
- **Fix:** In the non-replace_all branch, count occurrences first: if `before.split(oldStr).length - 1 > 1`, return an `ok:false` preview mirroring the executor's multi-match error instead of producing a misleading diff. Use the same replacen-equivalent (replace first only) only when exactly one match exists.

### [bug] readLines overflow drop keeps only the last ~MAX_BUF window, discarding already-complete earlier lines
- **Where:** `src/lib/stream-lines.ts:38-52`  ·  area: ts-agent-loop-rest  ·  confidence: high
- **Problem:** The overflow guard runs AFTER appending each read but BEFORE the inner line-splitting loop. `buf.lastIndexOf("\n", buf.length - MAX_BUF)` searches backwards starting at position `buf.length - MAX_BUF`, so it can only find a newline within the last MAX_BUF bytes and drops everything before it — including fully-formed, parseable NDJSON lines that simply hadn't been split out yet this iteration. For a server that emits one legitimately huge line (e.g. a multi-MB tool_call arguments blob from a cloud model) interleaved with normal lines, complete content/tool_call lines that arrived earlier in the same buffer get silently dropped before processLine ever sees them. The diag log fires, but real streamed data is lost.
- **Fix:** Drain all complete lines (run the indexOf split loop) BEFORE applying the overflow truncation, so only a genuinely unterminated oversized tail is dropped. Move the MAX_BUF check to after the inner while-loop, operating on the residual tail only.

### [perf] read_file re-reads the ENTIRE file from disk on every paginated call (O(n²) read amplification)
- **Where:** `src-tauri/src/agent/fs.rs:553-589`  ·  area: rust-fs  ·  confidence: high
- **Problem:** read_file always does `tokio::fs::read(&resolved)` to slurp the whole file into memory, then slices out the requested `[start..end]` window. For any file below BINARY_SIZE_THRESHOLD (512 KiB) but larger than the 64 KiB page cap, an agent paginating with `next_offset` re-reads the full file from disk on every call. A 512 KiB file read in 64 KiB pages costs 8 full-file reads = ~4 MiB of disk I/O + 8 full allocations to return 512 KiB of content. Worst case is quadratic in the number of pages. The binary-detection head-scan only needs the first 8 KiB, and the returned slice only needs `[start..end]`.
- **Fix:** Open the file and seek: read only the head (8 KiB) for the binary check on the first page, then `seek(start)` + read at most `cap` bytes for the requested window. Use `tokio::fs::File` + `AsyncSeekExt`/`AsyncReadExt`, or a blocking `File::read_at`/`seek` in spawn_blocking. total_bytes comes from the already-stat'd `meta.len()` rather than `bytes.len()`. This makes each page O(page) instead of O(file).

### [perf] Watcher debounce map (`last_seen`) grows unbounded for the life of the watch
- **Where:** `src-tauri/src/agent/fs_watcher.rs:238-273`  ·  area: rust-fs  ·  confidence: high
- **Problem:** When `debounce_ms > 0`, every distinct event path is inserted into the per-watch `last_seen: HashMap<String,(u64,String)>` and entries are NEVER removed or expired. A recursive watch over an active tree (build output, node_modules churn, log rotation, a `git checkout` touching thousands of files) accumulates one heap-allocated String key per unique path seen, retained until the watcher is stopped or GC'd (up to 30 minutes). On a busy repo this is an unbounded memory leak proportional to the number of distinct paths touched, not to the live file count.
- **Fix:** Prune stale entries opportunistically inside the callback: on insert, if the map exceeds a cap (e.g. 4096) or every Nth event, drop entries whose `last_ts` is older than the debounce window (`ts - prev_ts >= debounce`). Those entries can never debounce again, so removing them is free of correctness cost. Alternatively cap the map size with an LRU-style eviction.

### [perf] MoA proposers run with identical params — no diversity, parallel calls wasted
- **Where:** `src/lib/workflow/nodes.ts:244-250`  ·  area: ts-workflow  ·  confidence: high
- **Problem:** runMoa fans out N proposers via `runSub(ctx, { stream: false })` with NO per-member variation — same model, same params, same (default) temperature, same prompt. Mixture-of-Agents only works when the proposers diverge so the synthesis pass has independent drafts to reconcile. The sibling runConsistency node explicitly fixes exactly this by varying temperature per member (`temperature: sampleTemperature(i, n)`, nodes.ts:281) with a comment (nodes.ts:274-276) stating that without it 'they collapse to near-identical text and the consistency vote has nothing to weigh.' MoA never received the same fix: nodes.test.ts asserts distinct temperatures for consistency (lines 202-223) but only asserts call COUNT for MoA (lines 147-155). Net effect: every MoA card pays for N parallel model generations (up to 8) that produce ~identical output, then a synthesis pass over duplicates — wasted compute and a degraded result vs. the node's whole purpose.
- **Fix:** Vary the proposers the same way consistency does — pass a per-index temperature: `runSub(ctx, { stream: false, temperature: sampleTemperature(i, n) })`. sampleTemperature already exists and is exported within the module; reuse it so MoA and consistency can't drift. Optionally add a nodes.test.ts assertion that the N MoA proposer calls carry distinct temperatures, mirroring the consistency test.

### [perf] RoundtableView re-parses + re-sanitizes markdown on every bubble render (no cache)
- **Where:** `src/components/RoundtableView.tsx:67-69 (RtTurnBubble body); also 1349-1351 and 1483-1485 (transcript maps)`  ·  area: react-perf  ·  confidence: high
- **Problem:** RtTurnBubble renders done turns with `dangerouslySetInnerHTML={{ __html: renderMarkdown(t.text) }}` — calling the full marked -> highlight.js -> DOMPurify pipeline directly in render with NO cross-render cache. The memo on RtTurnBubble only helps while object identity is preserved during a live run; the moment any done bubble re-renders (theme toggle, the outcome viewer at line 1483, the outcomes list, a remount on view-switch, or a single non-identity-preserving flush), every visible bubble re-runs the entire parse+sanitize. MessageList solved this exact problem with the module-level `cachedMarkdown` Map (MessageList.tsx:69-81) keyed on the content string, but RoundtableView does not share or reuse it. A 30-turn transcript with code blocks pays the full sanitize cost N times on each such re-render.
- **Fix:** Export `cachedMarkdown` from a shared module (or lift it out of MessageList) and call it here instead of `renderMarkdown` for done turns. Strings are immutable so a completed turn's HTML is parsed once and reused across every subsequent render, the same FIFO-bounded cache MessageList already relies on.

### [perf] katex (~270 KB JS) is NOT routed to the markdown chunk and lands in the main boot bundle
- **Where:** `vite.config.ts:60-68`  ·  area: build-boot  ·  confidence: high
- **Problem:** The manualChunks function splits highlight.js, marked, and dompurify into a 'markdown' chunk, but omits katex. katex is statically imported by src/lib/markdown.ts (`import katex from "katex"`). ### [perf] markdown chunk is eagerly <link modulepreload>-ed at boot — the 'renders before markdown' premise is false
- **Where:** `vite.config.ts:32-38`  ·  area: build-boot  ·  confidence: high
- **Problem:** The comment claims 'the empty chat shell renders before any markdown ever needs to highlight, so isolating them lets the main chunk land faster on first paint.' But markdown.ts is reached through a fully STATIC import chain: App.tsx line 55 `import { ChatWindow }` (not lazy) -> ChatWindow.tsx line 16 `import { MessageList }` (not lazy) -> MessageList.tsx line 17 `import { renderMarkdown } from "../lib/markdown"`. - **Fix:** To actually defer markdown rendering, make MessageList import renderMarkdown lazily (dynamic import on first non-empty transcript) or lazy-load MessageList itself in ChatWindow. Otherwise drop the misleading comment — the chunk split currently buys an extra HTTP request, not a faster first paint.

### [perf] QuickPrompt and DetachedChatView windows load the entire App bundle (full ChatWindow/markdown/katex tree)
- **Where:** `src/main.tsx:14-19`  ·  area: build-boot  ·  confidence: high
- **Problem:** main.tsx statically imports all three window roots — App, QuickPrompt, DetachedChatView — at module top, then branches at runtime. Because the imports are static (not React.lazy/dynamic import), the bundler cannot tree-shake App out of the QuickPrompt entry: there is one entry (index.html -> index chunk) shared by all three windows. ### [security] applescript_run output is NOT injection-fenced, unlike every other subprocess channel in these files
- **Where:** `src-tauri/src/agent/extras.rs:44-50`  ·  area: rust-shell-sandbox  ·  confidence: high
- **Problem:** applescript_run returns osascript stdout/stderr directly via String::from_utf8_lossy(...).into_owned() with NO injection_scan::scan_and_wrap. Every sibling subprocess output channel in this codebase (run_shell, run_code, format_code, diff_files, list_processes, kill_process stderr, clipboard_get, read_pdf) is fenced before re-entering the agent loop precisely because it is untrusted ingress. AppleScript can read arbitrary app data, web page contents (via Safari `do JavaScript`), Mail/Messages bodies, etc. — a prime injection vector — yet its output bypasses the fence entirely. This is an inconsistency the just-completed audit explicitly closed for the other channels.

### [bug] Keychain backend + file-fallback secrets become unreadable in test mode (get returns None unconditionally)
- **Where:** `src-tauri/src/settings.rs:352-355 (keychain_get) vs 304-309 (keychain_set)`  ·  area: rust-secrets  ·  confidence: high
- **Problem:** keychain_disabled() is true whenever FROGLIPS_SETTINGS_DIR is set. keychain_set() in that mode returns true as a no-op WITHOUT writing the key anywhere (lines 305-309: "Test mode treats success as 'do nothing'"). But keychain_get() in the same mode returns None unconditionally (lines 353-355). The two are asymmetric: set claims success but stores nothing, and get can never retrieve anything. The comment on keychain_set says the disk path keeps its plaintext, but keychain_get never consults the file in test mode, so any code path that resolves a secret via keychain_get (custom_backend.rs:206/230, mcp/mod.rs:351, OpenRouter key) silently gets None under FROGLIPS_SETTINGS_DIR even when a real key is present in settings.json/secrets.json. This means the integration test surface can never exercise real secret-resolution, masking regressions; and if FROGLIPS_SETTINGS_DIR is ever set in a non-test/dev launch (it is the only knob the diagnostics/temp tooling uses) every API key resolves to None at runtime.
- **Fix:** Make the two symmetric. In test/disabled mode keychain_get should fall through to the 0600-file backend (load_secrets().get(account)) rather than returning None, and keychain_set should actually write to that file — so the round-trip works. Or gate test behavior on a dedicated FROGLIPS_TEST_NO_KEYCHAIN var distinct from FROGLIPS_SETTINGS_DIR so production never trips the no-op path.

### [bug] Secret migration on keychain_get can lose the key copy if Keychain write succeeds but file rewrite fails
- **Where:** `src-tauri/src/settings.rs:366-380`  ·  area: rust-secrets  ·  confidence: high
- **Problem:** In keychain_get's migration path, after the Keychain set_generic_password succeeds it reloads the map and calls write_secrets to purge the file copy. ### [bug] Poisoned-mutex .unwrap() permanently bricks the MCP registry browse command
- **Where:** `src-tauri/src/commands/mcp.rs:266, 285-288`  ·  area: rust-net  ·  confidence: high
- **Problem:** mcp_registry_search locks REGISTRY_CACHE with .lock().unwrap() in three places. std::sync::Mutex poisons on panic: if any holder of this lock ever panics (e.g. an OOM or a future bug while the guard is held), every subsequent call to .lock().unwrap() panics too, so the MCP Browse tab is dead for the rest of the process lifetime. The cache is a pure performance optimization — a lock failure should degrade to a direct fetch, never crash the command.

### [bug] web_search parses a body that read_capped may have truncated mid-result
- **Where:** `src-tauri/src/agent/web.rs:375-422`  ·  area: rust-net  ·  confidence: high
- **Problem:** web_search reads the DuckDuckGo HTML with read_capped at WEB_FETCH_MAX_BYTES (1 MiB) and discards the `truncated` flag, then runs RESULT_RE over whatever bytes were captured. The DDG results page can exceed 1 MiB, in which case the final result(s) are silently cut off — and if the cut lands mid-<a> the last capture fails entirely. ### [bug] Scheduler emits trigger without checking shutdown between snapshot and emit loop
- **Where:** `src-tauri/src/workflows.rs:944-1019`  ·  area: rust-workflows  ·  confidence: high
- **Problem:** scheduler_scan iterates all rows and emits workflow-trigger events; the loop can be long if many workflows have many scheduled cards. ### [bug] HF model size cache keyed on top-level dir mtime returns stale sizes after pulls
- **Where:** `src-tauri/src/models.rs:194-209`  ·  area: rust-inference  ·  confidence: high
- **Problem:** cached_dir_size() invalidates the size cache only when the mtime of the top-level `models--org--name` directory changes. But HuggingFace's cache layout puts all weights under fixed subdirs (`blobs/`, `snapshots/<rev>/`, `refs/`) that already exist. ### [bug] dir_size aborts the entire MLX listing if any single subdirectory is unreadable
- **Where:** `src-tauri/src/models.rs:211-243`  ·  area: rust-inference  ·  confidence: high
- **Problem:** dir_size() uses `?` on `std::fs::symlink_metadata(&p)?` and `std::fs::read_dir(&p)?` while walking the tree. A single unreadable entry (permission error, a directory removed mid-walk by a concurrent HF pull/GC, or a broken mount) makes the whole function return Err. cached_dir_size then falls into the `dir_size(path).unwrap_or(0)` branch and caches 0 for that model. Worse, on first sight where mtime is unavailable, cached_dir_size returns the raw Err which list_mlx_models swallows with `.unwrap_or(0)` per-entry — but the partial total already accumulated is discarded, so a model with one transiently-locked file shows 0 bytes. Directory trees being walked while HF is actively writing/GC-ing is the common case here.
- **Fix:** Make the walk resilient: replace the `?` operators with best-effort handling (skip entries whose symlink_metadata/read_dir fails and continue accumulating) so one locked or vanished file yields a slightly-low total instead of 0/Err. e.g. `let Ok(md) = std::fs::symlink_metadata(&p) else { continue };` and `if let Ok(rd) = std::fs::read_dir(&p) { for entry in rd.flatten() { stack.push(entry.path()); } }`.

### [bug] list_ollama_models holds the PLMutex across child.wait() in the reader thread, serializing kill-on-timeout
- **Where:** `src-tauri/src/models.rs:57-88`  ·  area: rust-inference  ·  confidence: high
- **Problem:** The Child is wrapped in `Arc<PLMutex<Child>>` and the reader thread calls `child_for_reader.lock().wait()` — holding the parking_lot mutex for the ENTIRE duration of `wait()` (i.e. until the ollama process exits). On the timeout path the outer thread does `let mut c = child.lock(); c.kill();` — but the reader thread is still inside `wait()` holding that exact lock, so the outer thread blocks on `child.lock()` and can never deliver the kill until the child exits on its own. The timeout-kill is therefore a no-op against a genuinely hung daemon, defeating the stated purpose of the design ('kill the hung child so the reader thread can drain and exit').
- **Fix:** Don't hold the mutex across wait(). Either take the child out from behind the mutex into the reader thread entirely and signal the parent via the channel, or kill via the raw pid: capture `child.lock().id()` up front and on timeout send SIGKILL by pid (libc::kill) without taking the lock the reader is holding. Simplest: have the reader thread own the Child outright and only the kill path use a separate stored pid.

### [bug] task_create concurrency cap check is not atomic with insert (TOCTOU)
- **Where:** `src-tauri/src/task_queue.rs:82-154`  ·  area: rust-ipc  ·  confidence: high
- **Problem:** The active-task count is computed under the TASKS lock (lines 82-91) but the lock is then released; the spawn + map insert happens later (lines 124-154) under a fresh lock acquisition. Between the count and the insert the map is unlocked, so if two task_create calls ever interleave (e.g. if a future refactor makes the command async, or Tauri runs sync commands off-main, or an internal caller spawns from another thread) both can observe `active < MAX_CONCURRENT_TASKS` and both insert, exceeding the cap. It happens to be safe today only because the command is a synchronous #[tauri::command] serialized on the main thread — an implicit, undocumented invariant.
- **Fix:** Hold a single TASKS lock guard across the count-check and the insert (build the entry first, then under one lock: re-count, bail if full, else insert). That makes the cap correct regardless of how the command is scheduled.

### [bug] append_diag_log truncation rewrites file non-atomically, racing concurrent appends
- **Where:** `src-tauri/src/commands/misc.rs:146-169`  ·  area: rust-ipc  ·  confidence: high

- **Where:** `src-tauri/src/mcp/mod.rs:1004-1031`  ·  area: rust-net  ·  confidence: high
- **Problem:** is_blocked_ip (used for remote MCP URLs) and the identical custom_ip_blocked (custom_backend.rs:278) only block link-local, unspecified, multicast, and the single Alibaba IP 100.100.100.200. They do NOT block the limited-broadcast address 255.255.255.255, the 0.0.0.0/8 'this network' block beyond the single unspecified address, or the reserved 240.0.0.0/4 / 198.18.0.0/15 / 192.0.0.0/24 ranges. web.rs's is_safe_v4 (web.rs:47-68) explicitly blocks all of these (audit A29). The loopback/RFC1918 allowance is intentional for local MCP servers, but broadcast and reserved/future ranges are never a legitimate MCP endpoint and the divergence from the hardened web.rs guard is an inconsistency a future attacker-influenced discovery URL could probe.
- **Fix:** Factor the v4 reserved-range logic out of web.rs::is_safe_v4 and share it (minus the intentionally-allowed loopback/RFC1918) with is_blocked_ip and custom_ip_blocked so all three SSRF guards agree. At minimum add `|| v4.is_broadcast() || v4.octets()[0] >= 240`.

### [security] Remote MCP server URL is logged verbatim — embedded userinfo credentials leak to diagnostics
- **Where:** `src-tauri/src/mcp/mod.rs:1150-1154`  ·  area: rust-net  ·  confidence: high
- **Problem:** start_remote_server logs the full URL into the diagnostics ring and the structured `url` field. validate_remote_url accepts URLs with userinfo (e.g. https://user:secret@host/mcp — reqwest::Url permits it and host_str() still works), so a credential embedded in the endpoint URL is written in cleartext to the in-app diagnostics buffer. The rest of this module is meticulous about never logging secrets (stderr is length-only, tool params are dropped), so this is an inconsistent leak path.
- **Fix:** Strip userinfo before logging: build a display URL with `url.set_username("")` / `set_password(None)` (or log only scheme+host+path), and consider rejecting non-empty userinfo in validate_remote_url since bearer/OAuth is the supported auth path.

### [security] Remote MCP teardown DELETE is not pinned to the validated address set
- **Where:** `src-tauri/src/mcp/mod.rs:513-531`  ·  area: rust-net  ·  confidence: high
- **Problem:** On shutdown the remote transport sends a best-effort DELETE to r.url using r.client. r.client is the SSRF-pinned client built at start (good), so the request still lands on the originally-validated address — this is actually safe. The residual concern is only that the bearer token (r.auth) is attached to this DELETE without the https/loopback downgrade check that send_rpc_remote and call_api enforce elsewhere; if the configured MCP URL is plaintext http to a non-loopback host (validate_remote_url permits http to public hosts), the bearer travels in cleartext on teardown just as it does on every normal RPC. The real gap is upstream: validate_remote_url allows bearer-auth'd MCP over plaintext http to any host.
- **Fix:** Require https for remote MCP URLs that carry a token (or loopback for http), mirroring the secure_endpoint posture in oauth.rs — so a bearer credential is never sent over cleartext to a non-loopback host on any request path including teardown.
