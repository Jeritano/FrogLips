# Security Workflows — Research + Specs (2026-06)

Defensive/authorized security agent workflows for Froglips Flows. Five ship in the Flows gallery under the **Security** category: Security Auditor, Vuln Bug Hunter, Supply-Chain Sentinel, Exposure Monitor, Threat Model Crew.

All are defensive: audit your OWN code, test AUTHORIZED targets, monitor YOUR OWN exposure. Templates touching a live target or personal data open with an authorization gate (haltWhen authorized=no).

## Research patterns

### High-Recall SAST + LLM Verifier Filter (Two-Stage Triage)  (SAST / false-positive reduction)

- **What:** The dominant production pattern of 2026 (Semgrep AI-Powered Detection, QASecClaw, SAST-Genius). A deterministic high-recall scanner (Semgrep/CodeQL) casts a wide net and intentionally over-reports, then a coding-specialized LLM verifier examines each finding WITH surrounding source-code context to decide true-positive vs false-positive. The LLM never hunts from scratch — it does constrained verification of a concrete source-to-sink path, asking 'is this reported flow actually reachable and exploitable given sanitizers, validators, dead code, and test fixtures around it?' QASecClaw cut false positives 88.6% (560 to 64) on OWASP Benchmark and lifted precision from 0.695 to 0.951 with only a 3.1% recall cost. Critical safety detail: fail-OPEN — if the LLM times out or returns malformed output, RETAIN the finding rather than silently suppress it, so a flaky model never hides a real vuln.
- **Roles:** scanner (deterministic high-recall), verifier (coding-specialized LLM with code context), triage reporter
- **Tooling:** run_shell (semgrep --json / semgrep ci), read_file, search_files, find_references, workflow_set/get
- **Why it works:** Separating recall (scanner's job) from precision (LLM's job) beats LLM-only scanning on both axes — the scanner guarantees coverage and gives the model a precise anchor, so the LLM reasons about ONE concrete flow instead of hallucinating new ones. Context-aware filtering catches mitigations static analysis structurally cannot see (validation in a caller, unreachable dead paths, mock data). Fail-open prevents the verifier from becoming a silent vuln-hider.
- **Froglips mapping:** critic node is the spine. verifyCmd='semgrep --config auto --json --quiet <path>' RUNS the scanner each iteration and injects findings + exit code into the loop; criticSystemPrompt = skeptic/critic preset framed as 'constrained verifier — for each finding decide TRUE_POSITIVE or FALSE_POSITIVE using read_file/find_references on the named source and sink, justify with the actual mitigating context, NEVER invent new findings'; tools=[read_file,search_files,find_references]; passThreshold gates on the model confirming zero unmitigated true-positives remain. Bake fail-open into criticPrompt: 'if you cannot verify a finding, KEEP it and label NEEDS_HUMAN — never drop on uncertainty.' Use workflow_set to write the triaged finding list to scratchpad for the report card.

### Auditor / Critic Decomposition (GPTLens propose-then-refute)  (vulnerability discovery / FP reduction)

- **What:** GPTLens-style split of detection into two adversarial roles: AUDITOR agents propose candidate vulnerabilities (optimized for recall — find everything plausible), and a separate CRITIC agent refutes, ranks, and prunes them (optimized for precision — kill the noise). The critic argues AGAINST each finding, demanding a concrete exploit path; anything it can refute (sanitized input, type constraint, auth check upstream) is dropped. Extended forms add a courtroom of role-specific agents (security-researcher, code-author, moderator, review-board) that argue to a collective verdict, and Socratic Critic/Supporter/Judge debate to dampen single-model hallucination. Because auditor and critic are decoupled, the critic is not anchored on its own guesses, which is exactly what drives down the false-positive rate.
- **Roles:** auditor / proposer (high recall), critic / refuter (precision, argues against), ranker-judge (final verdict + severity)
- **Tooling:** read_file, search_files, find_references, git_log/git_blame (who introduced, when), workflow_set/get
- **Why it works:** A model grading its own output is biased toward confirming it. Giving refutation to an INDEPENDENT agent with an adversarial prompt removes that bias and forces every surviving finding to withstand a hostile reading — the same dynamic that makes human red-team/blue-team review work. Ranking by exploitability lets a human spend attention on the few findings that survived the gauntlet.
- **Froglips mapping:** Two-card chain or a critic node. Card 1 = auditor (researcher/coder preset, prompt 'enumerate ALL plausible vulns, favor recall, cite file:line for source and sink'), writes candidates via workflow_set('candidates',...). Card 2 = critic node (skeptic preset) whose criticPrompt loads scratchpad candidates and 'for each, attempt to REFUTE with concrete code evidence via find_references/read_file; drop refuted, keep + rank the rest by exploitability'. For the courtroom/Socratic variant use a moa node — parallel members = security-researcher / code-author / moderator personas, synthesis member = review-board rendering the collective verdict. haltWhen{key:'unresolved_high',equals:'0'} ends the flow once no high-severity finding survives refutation.

### Self-Consistency Vote on Ambiguous Findings  (FP reduction / variance control)

- **What:** For borderline findings where a single pass is unreliable, sample the SAME verification question N times in parallel at varied temperatures (research uses 3 calls at temp 1.0/0.7/0.9) and take a MAJORITY VOTE on the verdict (exploitable / not exploitable / severity). This collapses verdict variance on exactly the cases that produce flaky false positives and false negatives — the model that says 'vulnerable' once but 'safe' twice gets correctly classified as safe-by-majority. It is a cheap, fully-local precision booster applied selectively to the ambiguous tail rather than the whole corpus.
- **Roles:** N independent voter samples (same question, different temps), vote aggregator / tie-breaker
- **Tooling:** read_file, find_references, workflow_get (the ambiguous finding under vote)
- **Why it works:** LLM verdicts on ambiguous security questions are high-variance; one sample is a coin flip near the boundary. Majority voting is variance reduction — independent errors cancel, the stable signal survives. Applying it only to the contested tail keeps it cheap while removing the noisiest class of false positives (the ones that flip between runs).
- **Froglips mapping:** This IS the consistency node. After the SAST+verifier critic produces a 'contested findings' list in scratchpad, feed each into a consistency node: N parallel samples of the prompt 'Is finding X a real, exploitable vulnerability given this code? Answer EXPLOITABLE or SAFE with one-line justification', vote selects the verdict. Wire it as the precision gate downstream of the critic — only ambiguous findings (those the critic marked NEEDS_VOTE) get routed here, so you pay for N samples only on the tail. workflow_set the majority verdict back so the cascade and report cards read a clean list.

### Cascade Escalation for Hard / Inter-Procedural Vulns  (escalation / hard-reasoning)

- **What:** Most findings are dispatched cheaply by local models; the genuinely hard ones — deep inter-procedural taint flows, business-logic/IDOR/auth-bypass bugs, crypto-protocol misuse — need stronger reasoning. The cascade pattern keeps the bulk local and escalates ONLY the residual hard cases to a stronger model. This matches Semgrep's finding that LLM contextual reasoning is what catches IDOR and broken-authorization logic bugs that pattern-matching misses, and SemTaint/Artemis using LLMs specifically to resolve call edges and path conditions that static taint analysis cannot. Escalate on a confidence/severity threshold, not blanket — the economic doctrine of local-bulk, sparse-cloud-touch.
- **Roles:** local triager (bulk, cheap), escalation reasoner (hard inter-procedural / logic vulns)
- **Tooling:** read_file, find_references, search_files, run_shell (re-run scanner with deeper config on the escalated module only)
- **Why it works:** Vulnerability difficulty is heavy-tailed: a small fraction (multi-hop taint, auth logic, crypto) consumes most of the reasoning budget while the majority are shallow. Routing only the tail to a stronger model gives frontier-grade results on the hard cases at near-local cost. Logic/authorization bugs in particular are where local pattern matching fails and contextual reasoning pays off.
- **Froglips mapping:** cascade node, configured to the owner's flat-rate Ollama :cloud doctrine. escalateModel='glm-4.6:cloud', escalateBackend=ollama. Local stage handles per-finding triage; criticPrompt defines escalation trigger: 'escalate ONLY findings tagged HARD — inter-procedural taint crossing >2 functions, auth/IDOR/business-logic, or crypto misuse'. passThreshold tuned so shallow findings resolve locally and never touch cloud. The escalated reasoner gets read_file/find_references to trace the full source-to-sink path the scanner flagged but could not resolve. NEVER a per-token frontier API — flat-rate :cloud tag only, honoring the economy doctrine.

### Verified Secret Scanning Loop (scan -> live-verify -> confirm)  (secret scanning)

- **What:** Detection alone drowns teams in noise (test keys, expired tokens, example creds). The 2026 best practice is verification-FIRST: gitleaks/detect-secrets at the edge for speed, then TruffleHog with --only-verified in the deeper pass, which makes a live provider API call to confirm a candidate credential is ACTUALLY ACTIVE right now. An agent loop wraps this: run the scanner, then for each candidate the agent classifies (real-vs-fixture, scoped, blast radius), and only verified-live secrets become P0. AI-assisted commits leak secrets at ~2x baseline (3.2%), so this belongs in the agent's own commit/repo hygiene loop. Strictly defensive: scanning YOUR OWN repo/history for YOUR OWN leaked credentials.
- **Roles:** fast edge scanner (gitleaks), verifier (trufflehog --only-verified, live provider check), triager (blast-radius + rotation guidance)
- **Tooling:** run_shell (gitleaks detect / trufflehog filesystem --only-verified --json), git_log (history scan), read_file, ask_user (confirm scope before any live verification call)
- **Why it works:** Regex/entropy detection produces enormous false-positive volume; live verification collapses it to credentials that demonstrably work, eliminating alert fatigue. Splitting fast-edge from deep-verified gives both commit-time speed and pipeline-depth confidence. Pairing the scanner with an LLM triager adds blast-radius reasoning (what does this key unlock, rotate-now vs monitor) that raw tool output lacks.
- **Froglips mapping:** critic node with verifyCmd='trufflehog git file://. --only-verified --json' (or gitleaks for the fast pass). The loop runs the scanner, injects verified findings, and the critic (skeptic preset) triages each: real secret vs fixture, scope, rotation priority — writing a ranked list via workflow_set. Because live verification calls external providers, the flow's FIRST card MUST be an authorization gate (see Pattern 7) confirming the repo is the user's own; haltWhen blocks the scan card until scope_confirmed=='yes'. Keep all keys out of the model: any breach/provider lookups go through call_api with Keychain-injected keys, never pasted into prompts.

### Fuzzing-Guided Multi-Agent Discovery + Reproduction  (fuzzing / dynamic discovery)

- **What:** FuzzingBrain V2 (multi-agent on OSS-Fuzz: 90% detection on AIxCC 2025, 29 confirmed zero-days) shows the loop: LLM agents do code comprehension, synthesize seeds, generate harnesses, and triage crashes, with DETERMINISTIC tool-based verification closing the loop — a candidate vuln is only accepted once a fuzzer actually reproduces a crash. This gives ground-truth confirmation that text-only review can never reach: the bug is real iff there's a reproducing input. The agent then localizes (control-flow 'suspicious point' abstraction) and can draft a patch (AutoCodeRover-style, 61-72% plausible-patch rate on historical OSS-Fuzz bugs). Strictly for YOUR OWN code / authorized targets.
- **Roles:** harness + seed synthesizer, fuzzer runner (deterministic ground truth), crash triager / localizer, patch drafter (optional)
- **Tooling:** run_shell (build harness, run libFuzzer/AFL/cargo-fuzz, collect crashes), read_file, find_references, git_diff (propose patch)
- **Why it works:** A reproducing crash is incontrovertible — it eliminates the entire false-positive problem for the bugs it finds, because the verification is a real execution, not a model opinion. The LLM contributes where fuzzing is weak (writing harnesses, synthesizing structured seeds, explaining the root cause), and the fuzzer contributes where the LLM is weak (proving the bug actually fires). Deterministic tool verification is the throughline of every high-accuracy 2026 system.
- **Froglips mapping:** critic node where verifyCmd builds and runs the fuzzer (e.g. 'cargo fuzz run <target> -- -max_total_time=60' or an OSS-Fuzz harness) and injects the exit code + crash artifact. The model's job each iteration: write/repair the harness and seeds (shell preset + read_file/find_references), then read the crash to localize. passThreshold = a reproducing crash captured; maxIters bounds the budget. Escalate stubborn root-cause analysis or patch synthesis via cascade to glm-4.6:cloud. Gate behind authorization since fuzzing executes target code. budget nodeConfig (maxMs) caps fuzz time so the flow stays economical.

### Authorization / Scope-Confirmation Gate + Agent-Driven Threat Model  (governance / threat modeling)

- **What:** Two defensive bookends. FRONT: every workflow that touches a live target opens with a hard authorization gate — a card that REFUSES to proceed until the user explicitly asserts ownership/authorization of the asset (own repo, own domain, own host, own credentials, CTF, or written authorization). This is non-negotiable scope confirmation, enforced by a halt condition, before any scan/verify/recon/breach-lookup runs. BACK/parallel: STRIDE-GPT / tachi / Auspex-style agent threat modeling, where the LLM is a first-draft generator and completeness validator — it ingests architecture/diagram/code, enumerates STRIDE categories, builds attack trees, maps to MITRE ATT&CK/ATLAS, and a human keeps/discards/edits. The 2026 pattern is explicitly 'AI drafts, human decides' — never autonomous risk acceptance.
- **Roles:** authorization gatekeeper (refuses without asserted scope), threat-model drafter (STRIDE + attack tree), completeness validator, human keep/discard/edit
- **Tooling:** ask_user (assert ownership/authorization), read_file/list_dir/search_files (ingest architecture + code), web_search (CVE/technique enrichment), read_pdf (design docs), workflow_set (scope_confirmed flag)
- **Why it works:** A scope gate makes the defensive boundary structural rather than aspirational — the flow CANNOT proceed against an unauthorized target because the halt/gate blocks it, which is the right place to enforce 'authorized only.' For threat modeling, AI excels at exhaustive first-pass enumeration (the tedious completeness problem) while humans retain judgment on what's real and acceptable — combining them beats either alone and keeps a human accountable for risk decisions.
- **Froglips mapping:** The gate is the FIRST card of every live-target template: preset general, prompt 'Confirm you OWN or are AUTHORIZED to test this target. State the asset and your authorization basis (own repo/domain/host, CTF, or written authorization).', then ask_user, then workflow_set('scope_confirmed', answer). Use haltWhen{key:'scope_confirmed',equals:'no'} to halt, and gate downstream scan cards so they only run when scope_confirmed=='yes'. The threat-model body is best as a moa node: parallel members enumerate per-STRIDE-category (Spoofing/Tampering/Repudiation/Info-disclosure/DoS/Elevation) reading the codebase, synthesis member assembles the attack tree + MITRE mapping and flags gaps for human review. Pair with a critic node set to 'completeness validator' to catch missed assets/entry points before handing the human the keep/discard list.

### Personal Breach & Stealer-Log Sweep (HIBP self-monitoring)  (Credential / breach exposure monitoring)

- **What:** A self-monitoring flow that checks whether YOUR OWN email addresses and accounts appear in known data breaches, pastes, and infostealer (stealer-log) dumps via the Have I Been Pwned v3 API. The model is handed a list of the owner's own emails (from a scratchpad seed card or ask_user), then for each it queries breachedAccount, pasteAccount, and stealerLogsByEmail. Results are aggregated into a single exposure briefing: which breach exposed which account, what data classes leaked (passwords, tokens, addresses), and which accounts now show up in stealer logs (the highest-severity signal, because it means a device was infected). The HIBP key is registered once in the API registry and injected server-side, so the local model issues the lookups without ever seeing the 32-char key. Authorization is implicit (your own emails) but the first card still asserts ownership of every address before any lookup runs.
- **Roles:** AUTHORIZATION / scope card: ask_user to confirm every email/account being checked is owned by the operator; workflow_set('authorized', true); haltWhen authorized != true so nothing queries until ownership is asserted, researcher card with call_api: for each owned email, GET /breachedAccount/{email}?truncateResponse=false, GET /pasteAccount/{email}, GET /stealerLogsByEmail/{email} against the registered 'hibp' API; write raw JSON to scratchpad keyed by email, summarizer card: aggregate scratchpad into a severity-ranked exposure report (stealer-log hits first, then breaches with password/credential data classes, then pastes), with a remediation checklist (rotate password, enable 2FA, check device for infostealer)
- **Tooling:** call_api (registered 'hibp' API, auth_header 'hibp-api-key', auth_template '{key}', Keychain-injected), ask_user, workflow_set / workflow_get / workflow_keys, remember (persist the baseline exposure set for diffing on re-runs)
- **Why it works:** HIBP's breach/paste/stealer-log endpoints all require an authorized hibp-api-key header and only return data for the email you query, making this an inherently self-scoped defensive check. Froglips' call_api is purpose-built for exactly this: it injects the Keychain key into a configurable auth_header via the {key} template (so 'hibp-api-key: <key>' is set server-side), confines the request to haveibeenpwned.com, strips the auth header cross-host on redirects, and never exposes the key to the local model — so a local 8B model can run the entire sweep with zero credential exposure. The local-first economy doctrine holds: every lookup and the synthesis run on the local model; no cloud token spend.
- **Froglips mapping:** Plain 'researcher' card driving call_api in a loop over scratchpad-seeded emails, then a 'summarizer' card; gate the whole flow behind a haltWhen on workflow_set('authorized'). No cloud escalation needed — breach JSON synthesis is well within local-model reach.
- **Sources:** https://haveibeenpwned.com/api/v3, https://www.troyhunt.com/authentication-and-the-have-i-been-pwned-api/, https://haveibeenpwned.com/Subscription

### Pwned-Passwords k-Anonymity Self-Check (offline-safe credential validation)  (Credential / breach exposure monitoring)

- **What:** A privacy-preserving check of whether YOUR OWN passwords (or a vault export) appear in the Pwned Passwords corpus, using the k-anonymity range API. The password is SHA-1 hashed locally; only the first 5 hex chars of the hash are ever sent. The free range endpoint returns every 35-char suffix sharing that prefix plus a sighting count; the final match is done locally against the full hash, so the password — and even its complete hash — never leaves the machine. The flow optionally sends Add-Padding: true so traffic analysis can't infer a hit from response size. No API key is needed (this endpoint is unauthenticated), making it the safest possible self-check. Because the hashing and suffix comparison are deterministic string work, they are best done in a run_shell snippet or run_code sandbox, with the model orchestrating and interpreting the count.
- **Roles:** AUTHORIZATION / scope card: confirm the passwords/hashes belong to the operator's own accounts (ask_user); no third-party credentials, shell card (preset 'shell'): compute SHA-1 of each candidate locally, split prefix(5)/suffix(35), call_api or http_request GET https://api.pwnedpasswords.com/range/{prefix} with Add-Padding: true, grep the returned suffix list for a local match and emit the sighting count, summarizer card: produce a per-credential verdict (clean / pwned N times) and a rotation priority list, never echoing the plaintext password in the report
- **Tooling:** run_shell (sha1sum / openssl + grep — keeps plaintext local), call_api or http_request (range/{prefix}; no key required, Add-Padding header), workflow_set / workflow_get, ask_user
- **Why it works:** K-anonymity means the service never receives enough information to identify the password: only a 5-char SHA-1 prefix leaves the machine, the full comparison is local, and Add-Padding defeats size-based traffic inference. This is the canonical defensive self-check — Cloudflare and 1Password use the same model. Froglips keeps the plaintext inside the confined run_shell sandbox (approval-gated, cwd-confined) and only the harmless prefix touches the network via the SSRF-guarded http path; the local model orchestrates but never needs to see or transmit the secret. Unauthenticated endpoint = zero key management and zero cloud spend.
- **Froglips mapping:** A 'shell' preset card does the hashing + range lookup + local suffix match (deterministic, no model needed for the math), feeding counts into the scratchpad; a 'summarizer' card renders verdicts. Pure local, free API, no escalation.
- **Sources:** https://haveibeenpwned.com/api/v3, https://www.troyhunt.com/understanding-have-i-been-pwneds-use-of-sha-1-and-k-anonymity/, https://blog.cloudflare.com/validating-leaked-passwords-with-k-anonymity/

### Owned-Domain Breach & Stealer Monitor (org-wide exposure)  (Credential / breach exposure monitoring)

- **What:** For an organization you own/operate: monitor breach exposure across an entire domain rather than one email. With an HIBP subscription that has verified domain control, the flow queries breachedDomain/{domain} (every breached email address on the domain and the breaches involved), subscribedDomains (which domains are verified + their stats), and the stealer-log-by-domain endpoints (stealerLogsByEmailDomain / stealerLogsByWebsiteDomain) to surface infostealer infections affecting your users or customers logging into your site. The model diffs the current breached-address set against a stored baseline (remember/recall_memory) and reports only the NEW exposures since last run, turning a one-off audit into continuous monitoring. HIBP only returns domain results for domains you've proven control of, so the authorization is enforced by the provider itself.
- **Roles:** AUTHORIZATION / scope card: confirm operator controls the domain; call_api GET /subscribedDomains and assert the target domain appears as verified before proceeding; haltWhen the domain isn't in the verified set, researcher card with call_api: GET /breachedDomain/{domain}, /stealerLogsByEmailDomain/{domain}, /stealerLogsByWebsiteDomain/{domain}; persist the address->breach map to scratchpad, diff/summarizer card: compare against the baseline in memory (recall_memory), emit NEW-since-last-run breaches and stealer hits, ranked by data sensitivity; remember() the new baseline, blackboard card (optional): snapshot the run state so a later scheduled re-run can pick up the diff
- **Tooling:** call_api (registered 'hibp' API; Pro/subscription key, Keychain-injected), remember / recall_memory (baseline for diff-on-re-run), workflow_set / workflow_get, blackboard node (shared state across scheduled runs), ask_user
- **Why it works:** HIBP gates the breachedDomain and stealer-log-by-domain endpoints behind proven domain control and an authorized key, so the provider itself enforces 'your own assets only' — there is no way to run this against a domain you haven't verified. The stealer-log-by-domain endpoints are uniquely valuable for blue teams: they reveal customers/employees whose infected devices captured credentials for your login page, which a breach search alone misses. Froglips injects the subscription key server-side, the local model drives the queries and the diff, and remember/recall_memory turn it into stateful monitoring — perfect for a scheduled re-run that only alerts on deltas.
- **Froglips mapping:** 'researcher' card over call_api for the three domain endpoints, gated by a subscribedDomains verification card (haltWhen domain not verified); a diff card using recall_memory/remember for baseline state. Scheduled re-run via the app's task scheduler; local-only, flat subscription cost, no per-token spend.
- **Sources:** https://www.troyhunt.com/passkeys-k-anonymity-searches-massive-speed-enhancements-bulk-domain-verification-api/, https://haveibeenpwned.com/api/v3, https://support.haveibeenpwned.com/hc/en-au/articles/15316197244559-How-do-I-get-started-with-Domain-Search-monitoring

### Leaked-Secret Scan of Your Own Repos (pre-leak self-audit)  (Leaked-secret / code exposure monitoring)

- **What:** A defensive scan of YOUR OWN source code and git history for hardcoded secrets — API keys, tokens, passwords — before they leak or to catch ones already committed. The flow runs TruffleHog (800+ secret types with live verification) and/or Gitleaks over the local working tree and full history in the confined shell, optionally cross-checks GitHub's own secret-scanning alerts for your repos via the GitHub API, and uses code-reading tools to triage hits (is this a real live key, a test fixture, or a placeholder?). For any verified-live secret the report includes the file, the commit, and a rotation/revocation checklist. Because git history rewrites and revocation are destructive, those are recommendations only — the flow never auto-rotates. This is the same '5-minute exposure window' threat-actors exploit on public repos, run defensively against your own code.
- **Roles:** AUTHORIZATION / scope card: confirm the repos/paths are owned by the operator; restrict the scan root via workspaceRoot; ask_user before any history rewrite is recommended, critic node with verifyCmd: run 'trufflehog git file://. --only-verified --json' (and/or 'gitleaks detect --report-format json') as the verifyCmd; the exit code + JSON findings ground the critic's score so the loop continues until the tree is clean or maxIters hit, researcher card with call_api: GET the GitHub secret-scanning alerts endpoint for your own repos (/repos/{owner}/{repo}/secret-scanning/alerts) using a registered GitHub PAT, Keychain-injected, code-audit card: read_file / search_files / find_references on each hit to classify live-vs-fixture and write a triage verdict + remediation steps to scratchpad
- **Tooling:** run_shell (trufflehog, gitleaks — confined, approval-gated), critic node with verifyCmd (runs the scanner, injects exit code), call_api (registered 'github' API, PAT in Keychain, for secret-scanning alerts), read_file / search_files / find_references / git_* (triage), workflow_set / workflow_get
- **Why it works:** TruffleHog's --only-verified mode actually authenticates each candidate secret, so the verifyCmd's exit code is a high-signal pass/fail the critic node can score against — exactly the execution-grounded verification the critic node was built for (it runs the command every iteration and grounds the score in the real exit code). Running it locally on your own tree keeps the secrets on-device; the only network call is the GitHub secret-scanning API for your own repos, and that PAT is Keychain-injected so the local model never sees it. This is purely defensive (auditing code you own) and maps cleanly onto the critic-loop + call_api primitives with no cloud spend.
- **Froglips mapping:** A 'critic' node whose verifyCmd is the trufflehog/gitleaks invocation (exit code drives the loop), plus a 'researcher' call_api card hitting GitHub's secret-scanning alerts for your repos, plus code-reading cards for triage. Local critic model; optional :cloud cascade only for ambiguous triage reasoning.
- **Sources:** https://docs.github.com/code-security/secret-scanning/about-secret-scanning, https://www.theregister.com/2025/11/10/ai_companies_private_api_keys_github/, https://instatunnel.my/blog/github-secret-leaks-the-13-million-api-credentials-sitting-in-public-repos

### Own Attack-Surface Map (subdomains, certs, exposed services)  (Domain / attack-surface exposure monitoring)

- **What:** A passive external-exposure map of YOUR OWN domain: enumerate subdomains and forgotten/shadow-IT hosts from Certificate Transparency logs (crt.sh — free, no key) and historical-DNS providers (SecurityTrails), then enrich with what's externally visible per host (open ports, banners, exposed panels) via Shodan/Censys host lookups by IP. The goal is to find the assets you forgot you own — a stale admin subdomain, a dev box accidentally in a wildcard cert, an exposed database port — before an attacker does. Everything is passive and queries third-party indexes about your own domain; no active scanning of others. An explicit scope card pins the exact apex domain(s) and refuses any host that isn't a subdomain of an asserted-owned apex. Optional active validation (e.g. nmap) is allowed only against hosts the operator confirms they own/are authorized to test.
- **Roles:** AUTHORIZATION / scope card: ask_user to assert ownership of each apex domain; workflow_set('owned_apexes', [...]); every later card filters discovered hosts to subdomains of those apexes and drops anything else; haltWhen no apex asserted, researcher card: call_api/http_request GET https://crt.sh/?q=%25.{domain}&output=json for CT-log subdomains (no key); call_api to SecurityTrails (/v1/domain/{domain}/subdomains) and Shodan (/shodan/host/{ip}) with Keychain-injected keys; aggregate hosts->ports/banners in scratchpad, router node (optional): route each host by exposure type (open admin panel / database port / stale cert) to a tailored follow-up summary, summarizer card: rank hosts by exposure severity, flag forgotten/shadow assets and expiring/mis-issued certs, output a remediation backlog (decommission, firewall, renew, take offline)
- **Tooling:** call_api / http_request (crt.sh keyless; SecurityTrails + Shodan + Censys with Keychain-injected keys), run_shell (optional nmap/dig against operator-confirmed-owned hosts only), router node (classify hosts by exposure type), workflow_set / workflow_get / haltWhen (scope enforcement), ask_user
- **Why it works:** CT logs and passive-DNS/Shodan indexes are public, third-party datasets about your domain — querying them is non-intrusive reconnaissance of your own footprint, the standard blue-team attack-surface-mapping technique for finding shadow IT and forgotten subdomains. crt.sh needs no key (pure http_request); SecurityTrails/Shodan keys are Keychain-injected via call_api so the local model orchestrates without seeing them. The scope card + haltWhen + per-host apex filter are the critical defensive guardrail: the flow structurally cannot pivot to a domain the operator hasn't asserted ownership of, and any active probe (nmap) is gated behind explicit ownership confirmation and the approval-gated shell. Local model does enumeration and synthesis; no cloud spend.

### Pwned-Passwords k-Anonymity Self-Check (offline-safe credential validation)  (Credential / breach exposure monitoring)

- **What:** A privacy-preserving check of whether YOUR OWN passwords (or a vault export) appear in the Pwned Passwords corpus, using the k-anonymity range API. The password is SHA-1 hashed locally; only the first 5 hex chars of the hash are ever sent. The free range endpoint returns every 35-char suffix sharing that prefix plus a sighting count; the final match is done locally against the full hash, so the password — and even its complete hash — never leaves the machine. The flow optionally sends Add-Padding: true so traffic analysis can't infer a hit from response size. No API key is needed (this endpoint is unauthenticated), making it the safest possible self-check. Because the hashing and suffix comparison are deterministic string work, they are best done in a run_shell snippet or run_code sandbox, with the model orchestrating and interpreting the count.
- **Roles:** AUTHORIZATION / scope card: confirm the passwords/hashes belong to the operator's own accounts (ask_user); no third-party credentials, shell card (preset 'shell'): compute SHA-1 of each candidate locally, split prefix(5)/suffix(35), call_api or http_request GET https://api.pwnedpasswords.com/range/{prefix} with Add-Padding: true, grep the returned suffix list for a local match and emit the sighting count, summarizer card: produce a per-credential verdict (clean / pwned N times) and a rotation priority list, never echoing the plaintext password in the report
- **Tooling:** run_shell (sha1sum / openssl + grep — keeps plaintext local), call_api or http_request (range/{prefix}; no key required, Add-Padding header), workflow_set / workflow_get, ask_user
- **Why it works:** K-anonymity means the service never receives enough information to identify the password: only a 5-char SHA-1 prefix leaves the machine, the full comparison is local, and Add-Padding defeats size-based traffic inference. This is the canonical defensive self-check — Cloudflare and 1Password use the same model. Froglips keeps the plaintext inside the confined run_shell sandbox (approval-gated, cwd-confined) and only the harmless prefix touches the network via the SSRF-guarded http path; the local model orchestrates but never needs to see or transmit the secret. Unauthenticated endpoint = zero key management and zero cloud spend.
- **Froglips mapping:** A 'shell' preset card does the hashing + range lookup + local suffix match (deterministic, no model needed for the math), feeding counts into the scratchpad; a 'summarizer' card renders verdicts. Pure local, free API, no escalation.
