### Headline risks

1. **TOCTOU task-orphaning in `messaging/mod.rs::start()`** (Critical) — concurrent `start()` for the same channel leaks a running tokio task that can never be stopped. Real resource-leak under realistic UI double-click / concurrent-request conditions.
2. **Two unbounded memory leaks in the rate-limit map** (High ×2, same root cause) — expired per-sender entries are reset but never removed; the map grows O(every-sender-ever) for the life of a long-running desktop process.
3. **No network timeouts across IMAP and WebSocket I/O** (High ×2 + Medium) — `tls_connect`, IMAP search/fetch/store, and Slack/Mattermost WS reads can hang indefinitely on a silent/unresponsive server, stalling the entire channel task with no error surfaced.
4. **Silent inbound-message loss on gateway unmount** (High) — messages in-flight or queued when `useMessagingGateway` unmounts are dropped with no log and no reply; users never learn their message went unanswered.

The remote-execution security posture is **strong**: the SAFE_REMOTE_TOOLS allowlist, Computer-Use kill-switch, unattended deny-all gate, and serial-drain concurrency control were all independently verified as correctly implemented.

## 2. Findings by Severity

### CRITICAL

**C1. Concurrent `start()` calls bypass the task-liveness check, orphaning a running task**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/mod.rs:242-291` (race at L246-247 vs. insert at L289)
- *Why real:* The `task_alive()` check (L246) takes and releases its own lock, then L249-288 run with no lock held. A second concurrent `start()` for the same channel also sees `false`, spawns a duplicate task, and its insert at L289 overwrites the first handle — the first task runs forever and can never be reached by `stop()`. Reproducible via UI double-click or concurrent client requests.
- *Fix:* Acquire the tasks-map lock **once** and perform check + spawn + status-insert + handle-insert atomically inside that single critical section (or use `get_or_insert_with`-style guarding), so no other thread can interleave between the check and the insert.

**C2. SAFE_REMOTE_TOOLS allowlist is correctly enforced** — *verified-correct, no fix*
`/Users/joseph/Documents/Codex/local-llm-app/src/lib/agent-loop/runner.ts:1102, 1592-1612`
- *Why real (correct):* Double-gated — the advertised tool list is filtered by `Set.has()` (L1107-1109) and any runtime call outside the allowlist is rejected as `tool_not_allowed` (L1592-1612), with the allowlist precomputed as a Set (L1102). Enforcement always fires for messaging-gateway runs.
- *Fix:* None.

**C3. Serial message draining prevents concurrent runs on one conversation** — *verified-correct, no fix*
`/Users/joseph/Documents/Codex/local-llm-app/src/hooks/useMessagingGateway.ts:104-114`
- *Why real (correct):* `drainingRef` (guard at L105, cleared in `finally` L112) serializes `handleInbound`, preventing two concurrent agent runs from racing on the shared `histRef` conversation history.
- *Fix:* None.

### HIGH

**H1. Unbounded rate-limit `HashMap` growth — memory leak**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/mod.rs:146` (and root at L60, L144-157)
- *Why real:* Expired windows are reset to `(now, 0)` (L148) but the entry is never removed. Across many unique senders the map grows unbounded for the process lifetime — a genuine leak in a long-running desktop app. (Reported twice from two dimensions; same root cause as H2.)
- *Fix:* Evict on expiry (replace the reset with `rate.remove(&key)`), or run periodic cleanup removing entries older than `2 × RATE_WINDOW_SECS`, or switch to a bounded LRU / sliding window.

**H2. `rate` map keys never cleaned — entries leak on sender churn**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/mod.rs:60, 144-157`
- *Why real:* Same defect as H1 viewed from the async-leak dimension — `*e = (now, 0)` resets but never removes; map size is O(total unique senders ever).
- *Fix:* Same as H1 — fixing one fixes both. Track as a single work item.

**H3. Missing timeout on IMAP TCP connection and TLS handshake**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/email.rs:84-91`
- *Why real:* `tls_connect()` does `TcpStream::connect()` + `connector.connect()` with no timeout; both can hang forever on an unresponsive server, stalling the email poll loop (L166-177). Callers at L124 (validate) and L189 (poll) add no wrapper. The codebase already uses `tokio::time::timeout` in 26+ other spots — this is the outlier.
- *Fix:* Wrap both calls: `tokio::time::timeout(Duration::from_secs(10), TcpStream::connect((host,port))).await.map_err(|_| "TCP connect timeout")??;` and likewise for the TLS connect.

**H4. Missing timeout on IMAP operations (search, fetch, store, select, login, mark_seen)**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/email.rs:189-269` (also `mark_seen` ~L281)
- *Why real:* None of the session operations are timeout-wrapped; an unresponsive IMAP server hangs `poll_once()` indefinitely, blocking message processing with no error reported and violating the ~20s poll contract.
- *Fix:* Wrap each session op in `tokio::time::timeout(Duration::from_secs(20), …)` returning a descriptive timeout error, e.g. `… session.search("UNSEEN") …`.

**H5. Inbound messages queued/in-flight during component unmount are silently lost**
`/Users/joseph/Documents/Codex/local-llm-app/src/hooks/useMessagingGateway.ts:182-189` (cleanup L191-194)
- *Why real (re-rated medium→high):* Cleanup sets `disposed = true` and unlistens but does **not** await the in-progress `drain()` (started L185); any messages still in `queueRef`, or arriving after disposal, are dropped at L183 with no log and no reply. Silent data loss — users never learn their message was received but unanswered. Fix is straightforward yet entirely absent.
- *Fix:* Before completing cleanup, synchronously drain remaining queued messages (or persist them to survive unmount and drain on next mount). At minimum, `logDiag({level:'warn', source:'messaging', message: …queued messages will not be processed})` when discarding.

**H6. Computer Use disabled by default for remote runs** — *verified-correct, no fix*
`/Users/joseph/Documents/Codex/local-llm-app/src/hooks/useMessagingGateway.ts:150`
- *Why real (correct, re-rated critical→high):* Triple defense-in-depth — `computerUseEnabled:false` hardcoded (L150); `cu_*` excluded from SAFE_REMOTE_TOOLS so never advertised (system-prompt.ts:87, runner.ts:1592); hard execution gate blocks any `cu_*` call when the flag is off (runner.ts:1618), catching hallucinated tool names.
- *Fix:* None.

### MEDIUM

**M1. Status-insert / task-insert race window causes transient stale `running` flag**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/mod.rs:264-289`
- *Why real:* Status is inserted `running=true` (L264-276) and the lock released; the task handle is inserted later (L289). A `status()` call in the gap sees `running=true` but no task → `running: s.running && alive` reports `false` (L206). Brief but real "not running just after start" in the UI.
- *Fix:* Merge into a single lock acquisition that inserts both status and task atomically (folds into the C1 fix).

**M2. Missing untrusted-data fence on remote chat input**
`/Users/joseph/Documents/Codex/local-llm-app/src/hooks/useMessagingGateway.ts:134`
- *Why real (re-rated high→medium):* Remote messages from external platforms (Telegram/Discord/Slack/…) are injected as user messages without `fenceUntrustedData()`, unlike subagent answers (subagent.ts:16) and workflow handoffs (runner.ts:218) which both fence. These role-framing tokens (`<|im_start|>system`, `[INST]`, …) can materialize as real role boundaries. Downrated because allowlist + deny-all + iteration cap limit practical exploitability — but the fence would close a threat-model inconsistency.
- *Fix:* `import { fenceUntrustedData } from '../lib/agent-loop/untrusted-fence'` and wrap: `content: fenceUntrustedData(m.text, 'remote-chat')` when building the user message.

**M3. Missing validation for `messaging` config in `settings_set`**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/commands/misc.rs:629-803` (key allowed at L618)
- *Why real:* `messaging` is in `ALLOWED_SETTINGS_KEYS` but has no block in `validate_settings_patch()`, unlike mcp_servers/custom_backends/saved_apis/user_profile. Serde catches raw type errors, but there's no *semantic* validation — enabled channels can persist with empty `allowed_user_ids`, missing `imap_host`/`smtp_host`/`homeserver`/`server_url`/`username`, or out-of-range ports, failing only later at startup (messaging/mod.rs:250).
- *Fix:* Add a `messaging` validation block (after the saved_apis block, before `Ok(())`): object shape, per-channel required non-empty strings when enabled, `allowed_user_ids` an array of non-empty strings, ports valid `u16`. Mirror the existing validation blocks.

**M4. Discord heartbeat task leak on multiple op 10 (Hello)**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/discord.rs:139`
- *Why real:* `hb_task = Some(tokio::spawn(...))` is reassigned without aborting any existing heartbeat; a second op 10 in one `run_once()` (malformed stream) leaks the old heartbeat, which keeps sending. Low likelihood, real impact.
- *Fix:* Before L139: `if let Some(h) = hb_task.take() { h.abort(); }`.

**M5. Discord heartbeat task silently exits without status update**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/discord.rs:139-149`
- *Why real:* If the heartbeat loop breaks or panics, the main loop only learns indirectly via Discord's ~45s close-frame timeout — a blind window where the heartbeat is dead but the app looks connected; panics are unhandled.
- *Fix:* Share an `Arc<AtomicBool>` health flag the heartbeat sets on exit so `run_once()` detects death immediately; wrap the task body to log/report failure.

**M6. Matrix `since` / Telegram `offset` cursors not persisted across restarts**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/matrix.rs:108` (and telegram.rs ~L53)
- *Why real:* Cursors live only in `run()`-local scope and reset on task restart — Matrix re-syncs from the initial filter; Telegram restarts at `offset=0` (definite replay risk). Email uses server-side UNSEEN and is unaffected.
- *Fix:* Persist cursors to settings/disk per channel (e.g. `messaging:matrix:cursor`), load on startup, update atomically after each successful sync.

**M7. No timeout on long-lived WebSocket reads (Slack, Mattermost)**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/slack.rs:140` (similar in mattermost.rs)
- *Why real:* `stream.next().await` has no timeout and no active heartbeat (unlike Discord). A silent TCP stall (alive socket, no frames) blocks the task forever; kernel keepalive won't necessarily catch it.
- *Fix:* Wrap `stream.next().await` in `tokio::time::timeout` (~60s); on expiry close + reconnect. Add a Socket-Mode-level ping/pong heartbeat.

**M8. `reqwest::Client` rebuilt on every Telegram `send()` call**
`/Users/joseph/Documents/Codex/local-llm-app/src-tauri/src/messaging/telegram.rs:118-121`
- *Why real:* The client is reused across chunks within one `send()` but rebuilt on every external `send()` (from `messaging_send`). `reqwest::Client` is meant to be reused (connection pool).
- *Fix:* Hold a shared client via `once_cell`/`lazy_static` per connector, or pass one in from the dispatcher.

**M9. Unattended deny-all gate works correctly** — *verified-correct, no fix*
`/Users/joseph/Documents/Codex/local-llm-app/src/hooks/useMessagingGateway.ts:154-157`
- *Why real (correct, re-rated critical→medium):* `requestConfirmation` is hardwired to `approve=false, reason='unattended_denied'`, a sound fallback for any tool reaching the dangerous-tool gate on an unattended run. (Note: SAFE_REMOTE_TOOLS are already stopped earlier by the allowlist gate, so this is a backstop, not the primary control — the finder's framing slightly overstates its role.)
- *Fix:* None.
